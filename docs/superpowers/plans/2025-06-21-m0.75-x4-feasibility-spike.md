# M0.75: X4 Capture/Apply Feasibility Spike — Plan

> **For agentic workers:** This is a **research spike**, not production code. The deliverable is a documented **go/no-go decision**, not a polished feature. Time-box it. Throwaway code is acceptable and expected.

**Goal:** Prove (or disprove) that X4's MD/Lua + SirNuke Named Pipes can support host-authoritative entity replication before committing to the full M1+ protocol stack.

**Why this is a hard gate:** The entire architecture assumes the host can *capture* live entity state and clients can *apply* it to render coherent proxy ships. If X4 cannot apply external position/state updates smoothly, the design must change. Spec reference: design doc § Client simulation stance.

**Prereqs:** M0 complete (bridge stdin↔WS echo works). Two X4 installs (two machines or two accounts) with SirNuke Mod Support APIs and Protected UI Mode disabled.

---

## The three questions to answer

| # | Question | Pass criteria |
|---|----------|---------------|
| Q1 | Can we **read** a player ship's position/rotation/velocity at 5–10 Hz from Lua/MD and push it over the pipe? | Stable 5–10 Hz sample stream with <100 ms capture latency, no UI stutter |
| Q2 | Can we **spawn a proxy ship** in a second instance and **continuously update** its position/rotation from external data smoothly? | Proxy follows source path with visually acceptable interpolation; no teleport-snapping at 5–10 Hz |
| Q3 | Can we **suppress or tolerate** the client's local NPC sim in an active sector so host snapshots are authoritative? | Documented mechanism OR documented acceptable-divergence boundary |

### Task 1: Capture spike (Q1)

**Files (throwaway):**
- Create: `extension/spike/md/x4mp_spike_capture.xml`
- Create: `extension/spike/ui/x4mp_spike_capture.lua`

- [ ] **Step 1: Hook a periodic timer**

MD cue on `checkinterval` ~0.1–0.2s (or Lua `SetScript`/frame callback) that reads the player-controlled ship.

- [ ] **Step 2: Read ship state**

Capture via Lua `GetComponentData` / MD properties: position (x,y,z), rotation, sector, speed. Document exactly which API returns each value and its update rate.

- [ ] **Step 3: Emit over pipe**

Write one NDJSON line per sample to the bridge pipe. Bridge prints to stdout (reuse M0 stub).

- [ ] **Step 4: Measure**

Record actual sample rate and jitter over 2 minutes of flying. **Write findings to `docs/superpowers/notes/spike-q1-capture.md`.**

Pass: sustained 5–10 Hz, no frame hitching. If MD-only is too slow, note whether Lua frame hooks do better.

---

### Task 2: Apply spike (Q2)

**Files (throwaway):**
- Create: `extension/spike/md/x4mp_spike_apply.xml`
- Create: `extension/spike/ui/x4mp_spike_apply.lua`
- Create: `tools/spike-feeder/` (tiny Rust or script that replays Q1 samples into the 2nd instance's pipe)

- [ ] **Step 1: Spawn a proxy ship**

On second instance, spawn one ship (MD `<create_ship>` or equivalent) in the player's sector to act as the proxy.

- [ ] **Step 2: Drive its position from pipe data**

Feed Q1's captured sample stream into instance 2. Each sample sets the proxy ship's position/rotation. Test two approaches and compare:
  - **(a)** Direct position set each sample (snap)
  - **(b)** Client-side interpolation between samples (smooth)

- [ ] **Step 3: Assess smoothness**

Fly the source ship; observe the proxy. **Write findings to `docs/superpowers/notes/spike-q2-apply.md`** including: which API moved the ship, max achievable update rate, whether interpolation is required, and any physics/collision side effects.

Pass: proxy visually tracks the source at 5–10 Hz + interpolation. Document the verdict honestly if it snaps/judders.

---

### Task 3: Local-sim suppression (Q3)

- [ ] **Step 1: Observe divergence**

Load the same save on both instances, run for 5 minutes, compare an NPC-heavy sector (ship counts, station storage). Quantify how fast they diverge. **Write to `docs/superpowers/notes/spike-q3-divergence.md`.**

- [ ] **Step 2: Test suppression levers**

Investigate whether any of these are feasible from MD/Lua for an active sector:
  - Removing/disabling locally-spawned NPCs and replacing with host-driven proxies
  - Pausing local economy/job ticks
  - Accepting divergence for NPCs but authoritative-syncing only player-relevant entities

- [ ] **Step 3: Document the boundary**

State what *can* be made authoritative vs what must be accepted as divergent in v1. This directly sets M1 and M5 scope.

---

### Task 4: Go/No-Go decision

**Files:**
- Create: `docs/superpowers/notes/spike-decision.md`

- [ ] **Step 1: Summarize Q1–Q3 results** with the measured numbers.

- [ ] **Step 2: Record the decision:**
  - **GO** — proceed to M1 as designed; note any sync-rate or scope adjustments.
  - **GO (revised)** — proceed but with documented scope changes (e.g. interpolation mandatory, NPC sync deferred).
  - **NO-GO** — architecture rethink required; capture which assumption failed and candidate alternatives.

- [ ] **Step 3: Update the design spec** § Client simulation stance and Milestones to match the decision.

## Done criteria
